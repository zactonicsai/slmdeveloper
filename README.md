# Complete SLM LoRA Java DTO Dev Environment

This project builds a local development environment for experimenting with small language models that generate Java DTOs using:

```
docker compose exec -e OLLAMA_BASE_URL=http://ollama:11434 -e OLLAMA_MODEL=java-dto-assistant slm-dev python scripts/query_ollama.py --prompt "Create a Java 21 DTO named OrderRequest with Lombok Builder, Swagger Schema examples, and Jakarta Validation."
```

- Python
- Hugging Face Transformers
- PyTorch
- PEFT LoRA
- FastAPI
- Ollama
- Docker Compose

The example dataset teaches the model Java DTO patterns with:

- Lombok
- Jakarta Validation
- Swagger/OpenAPI `@Schema`
- Java 21 style DTO conventions
- Safe API error responses

## Important note about Ollama and LoRA

This project has two model paths:

### 1. Immediate Ollama model

`java-dto-assistant` runs in Ollama using:

```text
qwen2.5-coder:0.5b
```

It is customized with a Modelfile system prompt for Java DTO generation.

### 2. True fine-tuned LoRA model

The Hugging Face training script saves a PEFT LoRA adapter to:

```text
outputs/java-dto-lora
```

That adapter runs through:

- `scripts/infer.py`
- `slm-api`

To run a fully fine-tuned model in Ollama, use this flow:

```bash
make train
make merge
```

Then convert the merged Hugging Face model to GGUF with llama.cpp, place the `.gguf` file in:

```text
models/
```

Then update:

```text
ollama/Modelfile.custom-gguf-template
```

And create your Ollama model:

```bash
ollama create java-dto-finetuned -f ollama/Modelfile.custom-gguf-template
```

## Troubleshooting Ollama connection errors

If `make ollama-test` shows a connection error, run:

```bash
docker compose ps
docker compose logs -f ollama ollama-setup
```

The first run can take time because Ollama must pull `qwen2.5-coder:0.5b` and create `java-dto-assistant`.

Use:

```bash
make ollama-ready
make ollama-test
```

Important network detail:

- From your host machine, Ollama is at `http://localhost:11434`
- From another Docker container, Ollama is at `http://ollama:11434`

The updated `make ollama-test` runs inside the `slm-dev` container and uses `http://ollama:11434`.

### If only `qwen2.5-coder:0.5b` appears but `java-dto-assistant` does not

Run:

```bash
make ollama-create
make ollama-tags
make ollama-test
```

Or manually:

```bash
docker compose run --rm -e OLLAMA_HOST=http://ollama:11434 ollama-setup
```

The setup container uses the Ollama CLI:

```bash
ollama create java-dto-assistant -f /workspace/ollama/Modelfile.java-dto-assistant
```

### If base model exists but `java-dto-assistant` is missing

Run:

```bash
make ollama-create
make ollama-test
```

Or manually:

```bash
curl -sS http://localhost:11434/api/create \
  -H "Content-Type: application/json" \
  -d @ollama/create-java-dto-assistant.json
```

The create JSON must use:

```json
{
  "model": "java-dto-assistant",
  "modelfile": "FROM qwen2.5-coder:0.5b\n..."
}
```

Older examples sometimes use `"name"`, but current Ollama API examples use `"model"`.

## Build the custom Ollama model

Start the stack:

```bash
docker compose up --build -d
```

Build the custom Ollama model from the Modelfile:

```bash
make ollama-build-custom
```

If you want to run the builder inside Docker instead of your host:

```bash
make ollama-build-custom-docker
```

Verify the model exists:

```bash
make ollama-tags
```

You should see:

```text
java-dto-assistant
qwen2.5-coder:0.5b
```

Test it:

```bash
make ollama-test
```

Manual commands:

```bash
curl -sS http://localhost:11434/api/pull \
  -H "Content-Type: application/json" \
  -d '{"name":"qwen2.5-coder:0.5b","stream":false}'

curl -sS http://localhost:11434/api/create \
  -H "Content-Type: application/json" \
  -d @ollama/create-java-dto-assistant.json
```
